오늘은 GitHub Pages에서 바로 공개할 수 있는 개인 포트폴리오 사이트를 정리했다. 목표는 **한 페이지 포트폴리오를 만들고, 그 위에 블로그를 덧붙여서 앞으로의 기록을 계속 남길 수 있는 구조**를 갖추는 것이었다. 결과적으로는 React + Vite 기반의 정적 사이트로 구성했고, GitHub Actions로 배포가 자동화되도록 구성했다.

이 글은 그 과정에서 어떤 결정을 했는지, 왜 그렇게 했는지, 그리고 앞으로 어떻게 확장할지에 대한 기록이다.

## 1) 왜 React + Vite였나

처음에는 순수 HTML로도 충분할 것 같았지만, 앞으로 블로그 글이 늘어나고 페이지가 더 다양해지면 **재사용 가능한 구조**가 필요했다. React는 구조적으로 콘텐츠를 정리하기에 좋고, Vite는 로컬 개발 속도가 빠르며 배포용 빌드도 간단하다. 

또한 블로그를 Markdown으로 관리하려면 글 목록/필터/상세 페이지를 구성할 수 있는 기본 라우팅 구조가 있어야 했고, 그 점에서 React Router를 사용하는 방향이 자연스럽다고 판단했다.

## 2) GitHub Pages 배포 방식

배포는 GitHub Actions로 자동화했다. `main` 브랜치에 push하면 Actions가 빌드를 수행하고 결과물을 Pages로 올리는 구조다. 

처음에는 `gh-pages` 브랜치를 고려했지만, GitHub Actions가 더 안정적이고 관리가 쉬웠다. 실제로 빌드 결과물을 직접 커밋할 필요가 없고, 배포 시점도 명확하게 기록된다.

## 3) 해시(#) 없는 라우팅

GitHub Pages는 서버 리라이트 기능이 없기 때문에 `/portfolio` 같은 경로로 직접 접근하면 404가 발생한다. 처음에는 HashRouter(`/#/portfolio`)로 해결할 수 있었지만, URL이 보기 좋지 않다는 점이 계속 아쉬웠다.

그래서 **BrowserRouter + 404 리다이렉트** 방식을 적용했다.

- `public/404.html`에서 잘못된 경로를 `sessionStorage`에 저장한 뒤 `/`로 리다이렉트
- `index.html`에서 `spa-redirect`를 읽어 원래 주소로 복구

이 구조 덕분에 이제 `/portfolio`, `/blog` 같은 경로가 깨끗하게 동작한다.

## 4) 네비게이션 구조

네비바는 고민 끝에 **“포트폴리오”와 “블로그” 두 가지 축**으로 단순화했다. 

- 포트폴리오는 기술 역량과 프로젝트를 보여주는 공간
- 블로그는 기록과 축적의 공간

블로그 안에서 전공/음악/여행을 카테고리로 나누는 방식이 더 자연스러웠다. 즉, 상단 메뉴에 전공/음악/여행을 따로 두기보다는, **블로그 자체가 콘텐츠 허브 역할을 하도록 구조를 바꿨다.**

## 5) 포트폴리오 페이지 구조 정리

포트폴리오는 다음 흐름으로 구성했다.

1. **소개/키워드 영역** – 나를 한 줄로 설명하고, 핵심 키워드를 명확하게 보여줌
2. **우선순위 영역** – 로보틱스 → 컴퓨터비전 → 임베디드 → 풀스택 순의 관심도 정리
3. **대표 프로젝트 5개 강조** – 가장 중요한 프로젝트는 카드 형태로 강조
4. **전체 프로젝트 라이브러리** – 나머지 프로젝트는 접어서 볼 수 있게 정리
5. **기술 스택/경험/활동/연락** – 이력과 역량을 구조화

이 구성은 “한 번에 읽을 수 있지만, 깊게 들어가려면 들어갈 수 있는” 구조를 목표로 했다.

## 6) 블로그 기본 기능

블로그는 아직 글이 많지 않지만, 기본 기능은 미리 준비했다.

- **검색**: 글 제목/요약/본문/태그 전체를 기준으로 필터
- **카테고리 필터**: 전공 대신 “개발”로 통일하고, 음악/여행은 확장 카테고리로 분리
- **태그 필터**: 블로그 성격에 맞는 세부 키워드 분리
- **읽는 시간 계산**: 대략적인 소요 시간 표시

글은 Markdown 파일로 관리하고, frontmatter를 통해 제목/날짜/카테고리/태그/요약을 적는다.

## 7) 앞으로 확장하고 싶은 방향

아직 시작 단계라 내용은 부족하지만, 구조는 이미 준비되어 있다. 
이제부터는 **내용을 꾸준히 쌓는 것이 핵심**이다.

앞으로는 아래 방향으로 확장하려고 한다.

- 로보틱스/컴퓨터비전 관련 구현 기록을 기술적으로 정리
- 프로젝트 별 문제 해결 과정과 선택한 설계 근거 기록
- 단순 결과물보다 **왜 그렇게 설계했는지**를 중심으로 남기기
- 블로그 카테고리를 너무 세분화하지 않고, 핵심 흐름을 유지

## 마무리

오늘 하루 동안 가장 중요했던 것은 **구조를 깔끔하게 잡는 것**이었다. 
포트폴리오와 블로그가 동시에 성장할 수 있도록, 기본 틀을 최대한 탄탄하게 만들었다.

이제는 내용을 채울 차례다. 오늘 정리해 둔 이 구조가 앞으로의 기록을 꾸준히 쌓는 기반이 될 거라 믿는다.
